# Scene 场景 · 考题（输出倒逼输入）

> 先合上文档独立作答。🟢概念 🟡机制 🔴架构/陷阱

---

## 一、概念题 🟢

1. Scene 模块自己加载场景资源吗？实际加载委托给谁？
2. 它维护哪三张状态名单？三者关系如何？
3. Scene 为什么用 List 而非单值存场景名？
4. Scene 模块存在的核心价值是什么（既然 Resource 已能加载）？
5. Scene 持有场景对象本身吗？它存什么？

---

## 二、机制题 🟡

6. `LoadScene` 前做了哪四重校验？分别防什么？
7. 场景从"加载中"迁移到"已加载"发生在哪里？
8. 加载失败时名单如何变化？为什么这步关键？
9. 卸载成功时哪两张名单被修改？
10. `Shutdown` 卸载所有场景时为什么跳过"正在卸载中"的？

---

## 三、架构 / 陷阱题 🔴

11. 为什么异步资源操作都需要"状态追踪层"？不追踪会出什么竞态 bug？
12. "Resource 管怎么加载、Scene 管能不能加载"体现了什么职责剥离原则？
13. 如果加载失败回调忘记从"加载中"移除，会发生什么？
14. 边加载边卸载同一场景为什么被禁止？四重校验如何拦截？
15. Scene 与 Sound 同为"委托 Resource + 状态追踪"模式，它们和直接调 Resource 有何不同？
16. 多场景叠加（Additive）加载时三态名单如何支持？

---

## 四、实操题 ✍️

17. 给精简版加 `ReloadScene`（先卸载再加载），注意异步时序与状态校验。
18. 实现"场景切换"：卸载当前所有场景 + 加载新场景，加载完成后回调通知。
19. 分析：连续调两次 LoadScene("A") 会怎样？为什么这是好的设计？

## 参考答案要点

<details>
<summary>点开核对</summary>

- **1**：不加载；委托 `IResourceManager.LoadScene/UnloadScene`。
- **2**：已加载/加载中/卸载中；三者互斥，一场景同时只在一张(或都不在=未加载)。
- **3**：支持多场景同时加载(Unity Additive 叠加)。
- **4**：提供"状态一致性"层——校验场景能否被加载/卸载，防止重复加载、加载卸载冲突等竞态。
- **5**：不持有场景对象；只存场景资源名(string)。
- **6**：卸载中→不能加载；加载中→不能重复；已加载→不能再加载；通过则加入加载中名单。
- **7**：加载成功回调里（从 Loading 移除、加入 Loaded）。
- **8**：从"加载中"移除；关键是否则场景永久卡在加载态，再也无法加载。
- **9**：卸载中(移除) + 已加载(移除)。
- **10**：避免对正在卸载的场景重复发起卸载。
- **11**：异步操作有时延，期间重复/冲突请求会导致加载两次、卸载正在加载的等竞态；状态追踪+入口校验拦截。
- **12**：单一职责——通用加载器不被特定业务(场景并发规则)污染，状态合法性由上层薄管理器负责。
- **13**：该场景 SceneIsLoading 永远 true，后续 LoadScene 全被校验拦截，无法再加载。
- **14**：加载卸载操作同一场景会状态紊乱；LoadScene 校验 SceneIsUnloading、UnloadScene 校验 SceneIsLoading 互相拦截。
- **15**：多了状态追踪与防重入；直接调 Resource 则无并发保护，易竞态。
- **16**：三张名单用 List 可容纳多个场景名，多个场景可同时处于已加载状态。

</details>
